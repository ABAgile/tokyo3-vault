mTLS principals map a client certificate's SPIFFE URI SAN or email SAN to vault authorization scope. Callers presenting a matching cert authenticate without a bearer token — they get the scope (project / env, optional read-only flag) configured below. Workload services typically register a SPIFFE ID; human admins with corporate PKI cards register an email SAN. The cert itself is never stored — only the SAN string and scope.

Register principal

SPIFFE URI (spiffe://…) for workload identity, or an email address for human admin certs. The format is auto-detected. Workload SANs are checked first at mTLS handshake; email SANs are the fallback.
Principals must be scoped — the picker shows only projects + envs you have access to. "All environments in <project>" requires project-level membership.
